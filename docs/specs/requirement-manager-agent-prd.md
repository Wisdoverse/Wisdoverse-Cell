## 1. 执行摘要

### 1.1 一句话描述

> **Requirement Manager Agent 是一个 AI 驱动的需求管理服务，能自动从会议记录中提取结构化需求，并支持团队协作确认。**

### 1.2 核心价值

| 指标 | 传统方式 | 使用 Agent 后 | 提升 |
|------|----------|---------------|------|
| 会议后整理时间 | 30-60 分钟 | < 1 分钟 | **98%** |
| 需求遗漏率 | ~15% | < 3% | **80%** |
| 需求确认周期 | 3-5 天 | < 24 小时 | **75%** |
| 冲突发现时间 | 开发中期 | 录入时 | **提前 2-4 周** |

### 1.3 关键成功指标 (KPIs)

| 指标 | 定义 | 目标 | 当前 |
|------|------|:----:|:----:|
| **提取准确率** | 正确需求数 / 总提取数 | > 90% | 85% |
| **召回率** | 正确需求数 / 实际需求数 | > 85% | 80% |
| **确认率** | 24h 内确认的需求占比 | > 80% | N/A |
| **用户满意度** | NPS 评分 | > 40 | N/A |
| **系统可用性** | 月度 SLA | > 99.5% | 99.9% |

## 2. 产品背景

### 2.1 业务背景

**目标团队**: 2 人创业团队（1 经营 + 1 兼职开发）
**主营业务**: 录音分析 SaaS + 智能硬件
**当前阶段**: 项目启动期，与付费客户对齐需求

### 2.2 问题陈述

| 问题 | 影响 | 严重程度 |
|------|------|:--------:|
| 会议记录分散在飞书、微信等多渠道 | 需求遗漏、重复沟通 | 🔴 高 |
| 手动整理耗时，延误开发 | 团队效率低下 | 🔴 高 |
| 需求变更追踪困难 | 版本混乱、返工 | 🟡 中 |
| 无法快速检索历史需求 | 重复讨论、上下文丢失 | 🟡 中 |

### 2.3 解决方案概述

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        传统方式 vs Agent 方式                            │
├──────────────────────────────┬──────────────────────────────────────────┤
│         传统方式              │              Agent 方式                  │
├──────────────────────────────┼──────────────────────────────────────────┤
│  1. 开完会                   │  1. 开完会                               │
│  2. 人工整理会议纪要          │  2. 会议纪要自动发送到系统               │
│  3. 手动提取需求点            │  3. AI 自动提取结构化需求                │
│  4. 录入到需求管理工具        │  4. 飞书卡片一键确认/拒绝                │
│  5. 逐一通知相关人            │  5. 自动发布事件，触发下游流程            │
│  6. 定期手动检查冲突          │  6. 实时检测重复/冲突需求                │
│                              │                                          │
│  耗时: 30-60分钟              │  耗时: < 1分钟                           │
│  准确: 依赖人                 │  准确: AI + 人工校验                      │
└──────────────────────────────┴──────────────────────────────────────────┘
```

### 2.4 竞品分析

| 产品 | 定位 | 优势 | 劣势 | 对比 |
|------|------|------|------|------|
| Jira | 通用项目管理 | 功能全面、生态完善 | 重、贵、无 AI | 我们更轻量、智能 |
| Linear | 现代需求管理 | UI 好、速度快 | 无自动提取 | 我们有 AI 提取 |
| Notion AI | 知识管理 + AI | 多用途、AI 辅助 | 不专注需求 | 我们更专业 |
| 飞书文档 | 文档协作 | 国内生态好 | 无结构化管理 | 我们深度集成 |

**差异化**: 我们是唯一能从会议直接自动提取、并在飞书内闭环确认的方案。

### 4.2 功能详情

#### F02: 需求提取 (核心功能)

**描述**: 使用 Claude API 从会议记录中提取结构化需求

**输入**:
- 会议内容文本 (必填)
- 来源 (feishu/upload/wechat)
- 会议日期 (可选)
- 参与者列表 (可选)
- 额外上下文 (可选)

**输出**:
```json
{
  "requirements": [
    {
      "title": "支持离线录音",
      "description": "设备在无网络环境下能本地存储录音，联网后自动上传",
      "category": "硬件",
      "priority": "high",
      "source_quote": "客户说'这个功能必须有，我们现场经常没网'"
    }
  ],
  "decisions": [
    {
      "content": "先做离线录音，再做实时转写",
      "decided_by": "客户方李总"
    }
  ],
  "open_questions": [
    {
      "question": "离线存储的容量上限是多少？",
      "context": "需要和硬件团队确认存储芯片规格"
    }
  ]
}
```

**处理规则**:
1. 只提取明确表达的需求，不推测
2. 同一需求的不同表述要合并
3. 区分"需求"和"讨论"
4. "必须"、"一定要"等词标记为 high 优先级
5. 模糊需求生成 open_question

**错误处理**:
| 场景 | 处理方式 |
|------|----------|
| LLM 调用失败 | 重试 3 次，指数退避 |
| 响应解析失败 | 返回空结果，记录日志 |
| 内容过长 | 截断并警告 |

#### F06: 冲突检测

**描述**: 两阶段检测新需求与已有需求的关系

**检测流程**:
```
新需求输入
    │
    ▼
┌────────────────┐
│  向量相似搜索   │  ← 快速，无 LLM 调用
│  Top 5, θ>0.6  │
└───────┬────────┘
        │
        ├── 无相似 ──▶ 返回 NEW
        │
        ├── 相似度 > 0.85 ──▶ 返回 DUPLICATE
        │
        └── 0.6 < 相似度 < 0.85
                │
                ▼
        ┌────────────────┐
        │  LLM 深度分析   │  ← 仅需要时调用
        │  判断关系类型   │
        └───────┬────────┘
                │
                ▼
        返回 NEW / UPDATE / CONFLICT + 建议
```

**关系类型**:
| 类型 | 说明 | 建议操作 |
|------|------|----------|
| NEW | 全新需求 | 直接创建 |
| DUPLICATE | 重复需求 | 合并或忽略 |
| UPDATE | 更新/细化 | 更新已有需求 |
| CONFLICT | 矛盾冲突 | 人工决策 |

### 5.2 异常流程

#### 5.2.1 LLM 提取失败

```
提取失败
    │
    ├── 重试 3 次 (指数退避: 1s → 2s → 4s)
    │       │
    │       ├── 成功 → 正常流程
    │       │
    │       └── 仍失败
    │               │
    │               ▼
    │       ┌────────────────┐
    │       │ 断路器打开     │ (连续 5 次失败)
    │       │ 60s 后半开探测 │
    │       └────────────────┘
    │               │
    │               ▼
    │       返回错误，通知用户:
    │       "需求提取暂时不可用，请稍后重试"
    │
    └── 发送告警到运维群
```

#### 5.2.2 用户操作冲突

```
用户 A 点击确认
同时
用户 B 点击拒绝
        │
        ▼
   数据库层乐观锁检查
        │
        ├── A 先到达 → A 成功，B 收到 "需求已被处理"
        │
        └── 版本不一致 → 拒绝后到的操作，提示刷新
```
